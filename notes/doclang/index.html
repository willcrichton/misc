Thoughts on Document Languages

1 Introduction

Document languages (doc¬≠langs) are programming languages whose purpose is to generate documents like articles, books, web pages, and so on. Doc¬≠langs include LaTeX, Markdown, PHP, and many others. The motivating question of this particular document is: what is the essence of a doc¬≠lang?

On the one hand, we ask this question out of curiosity. Doc¬≠langs have existed for decades, yet no one has tried to formalize the semantics of any doc¬≠lang. What unexpected connections, categories, and concepts might be found in the process of designing a core doc¬≠lang semantics? What truly differentiates the expressive power of LaTeX from Markdown Scribble, and Typst? Where might we find document languages in unexpected places?

On the other hand, we ask this question out of necessity. Web browsers have matured into semi-stable platforms for documents. Compared with PDF, the browser offers advantages in both presentation (styling, layout, etc.) and computation (reactivity, animations, etc.). Recent years have seen an explosion of interest in web doc¬≠langs that leverage these advantages. For example, recent industry doc¬≠langs include MDX, Markdoc, Quarto, and JupyterBook. Recent academic doc¬≠langs include Idyll and Living Papers.

These new languages enable the interleaving of computations and document fragments in a manner of unprecedented complexity in the history of doc¬≠langs. Consequently, we have found that these newer doc¬≠langs often have fragile semantics. For instance, Quarto supports variables for reusing data between code blocks, but a different and non-interoperable system of variables is used to share text between document blocks. Living Papers supports components that react to changes in runtime values, but only if those values are the JSON subset of data types. MDX supports recursive nesting of JSX React within Markdown, but not for the case of higher-order components.

In sum, the goal of this document is to articulate the essence of a doc¬≠lang for two reasons. One, we seek to shed light on the general relationships between different document languages. Two, we seek to establish a well-defined and compositional basis for modern doc¬≠langs that overcomes the challenges described above.

2.2 Trees

We can expand the space of doclangs by enriching the document model in one of two ways: (1) non-linear structure, and (2) reactivity. We'll start by focusing on the former. A basic hierarchical document usually has this structure:

Documents are organized into blocks b, which are either paragraphs of text or labeled sections with sub-documents. Text ranges are split up into sequences of inline elements, either plain strings or decorated strings like bolding. Inline elements are not permitted to hold block elements.

2.2.1 Tree Literals

Languages like Markdown are, in essence, the tree-identity doclang Didtree‚Äã whose program structure is exactly the document structure. All of Markdown's complexity is the parser, i.e. constructing d from a concrete syntax. (We will return to parsing and concrete syntax later.)

This document model is often enriched with cross-document references such as anchors and links:

Here, a section can be labeled with an anchor a, and the section can be linked-to via a. Note that without labels, all tree documents (and all string documents) are well-formed, i.e. there is a well-defined way to render the document. With labels, a link can refer to a non-existent anchor, and so a document d (as distinct from a program that generates the document) can now be ill-formed. Ill-formed documents are usually either rejected by an AST analyzer (as you might find in Pandoc) or cause unexpected interactions in the renderer (such as clicking a webpage link that has no effect).

2.3 Reactivity

Reactive documents change their structure in response to user interaction. We use an Elm-like architecture as a simple model of reactivity ‚Äî we assume that a document has a single global state that updates in response to a stream of events. Formally, a reactive document language R is an extension of a document language D. A reactive document program consists of an initial state einit‚Äã, a view expression eview‚Äã, and an update expression eupdate‚Äã.
