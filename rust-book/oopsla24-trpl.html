2.2 Question Discrimination via Correlation

Hard Question #1 illustrates how a question can be difficult but not interesting. That is, if a reader gets it wrong, that doesn't tell us much about whether they actually understand Rust itself. (We intentionally added it to the book more as a silly puzzle than a conceptual challenge.)

We are interested in which questions are indicative of a general understanding of Rust. That is, if readers answer these questions correctly, then we feel they have a solid grasp on Rust fundamentals overall. In psychometrics, this quality is usually called "discrimination", as in a question's ability to discriminate between high-ability and low-ability students. Question discrimination is valuable on two dimensions:

  1. It is useful to know whether a question is discriminating per se. For example, we can use discriminating questions to construct short tests that quickly and effectively screen for Rust expertise. This could be useful as a pre-test for a Rust course, or as an interview for a Rust job.
  2. It is useful to know the qualities that make a question discriminating (or not). These qualities tell us about the concepts that are most essential to Rust, and they can potentially inform the design of good quiz questions for other programming languages as well.

Ideally, we might compare quiz question performance with an ecologically-valid metric such as ability to work within a real Rust codebase. But we do not have such data. A common alternative in psychometrics is to compare performance on an individual question against performance on the entire set of questions. The simplest form of this comparison is the correlation (Pearson's $r$) of a person's answer to a question (i.e., 0 or 1) against their test score (i.e., in the interval [0, 1]).

A caveat: average scores are computed for "triers", but unlike a standard testing setup, most readers did not answer all questions. So a reader's average score is their score up to the point where they stopped answering questions, not until the end of the test.

Nonetheless, a preliminary interpretation suggests that:

  • Understanding of ownership is, in fact, a discriminating skill for understanding Rust.
  • Questions about the purpose and concepts of ownership (undefined behavior, incompleteness) are far more discriminatory than questions about the mechanics of ownership (lifetimes, compilation failures).

More broadly, this suggests a prescription: those seeking to write quiz questions about type systems should consider limiting the number of questions that ask readers to "simulate the type checker" in their heads, especially questions that may be seen as a "gotcha" (you thought this would compile, but it actually doesn't for a subtle reason!). A more useful kind of question is to tell readers that something does or doesn't compile, and ask a more conceptual question about the reasons why or how.

2.3 Question Discrimination via IRT

Correlation provides a "coarse-grained" metric of question discrimination, in the sense that it does not account for the difficulty or guess-ability of the question. Another common model for question discrimination comes from item response theory (IRT). The high-level idea of iRT is to model a question as a function that takes as input some features of the reader and the item, and returns as output the probability that the reader will answer the item correctly. Or formally: for an item $i$ and a subject $j$ with features $Z^\mathsf{item}_i$ and $Z^\mathsf{subj}_j$, let $C_{i,j}$ be a random variable for whether $j$ answers $i$ correctly. IRT is a model for:

$$ P[C_{i,j} \mid Z^\mathsf{item}_i, Z^\mathsf{subj}_j] $$

There are a number of models in the literature, but a common one we use here is the "three-parameter logistic model". In this model, $Z^\mathsf{item}_i = [\alpha_i, \beta_i, \lambda_i]$ and $Z^\mathsf{subj}_j = [\theta_j]$ and the probability is modeled as:

$$ P[C_{i,j}] = \lambda_i + (1 - \lambda_i) * \sigma(\alpha_i * (\theta_j - \beta_i)) $$

These parameters are often interpreted as having the following meanings:

  • $\alpha_i$ is the discrimination factor. The higher $\alpha_i$ is, the steeper the sigmoid curve, meaning the more discriminative a question is at a given level of difficulty.
  • $\beta_i$ is the difficulty factor. The higher $\beta_i$ is, the lower $P[C_{i, j}]$ for a given $\theta_j$.
  • $\lambda_i$ is the baseline factor. It accounts for the probability of guessing.
  • $\theta_j$ is the ability factor. It reduces the entirety of a reader's understanding of whatever is tested by the quiz into a single number.

Note: in the actual paper, we probably want to give a short primer on interpreting item characteristic curves.

To use this model, we must learn the best values of $Z$ that fit the data. We follow prior work in using variational inference adapted from the py-irt library and built on the Pyro probabilistic programming language.

3 Application: Test Construction

As mentioned before, one application for these statistics is test construction. The Rust Book provides too large a pool of items to give on a single exam, but we can use these statistics to identify a small subset of questions that would be useful in discriminating between participants. We demonstrate three possible purposes and means of test construction.

3.1 Maximal Correlation Subset

One metric for test construction is to find a set of items whose average score is maximally correlated with overall test performance. In theory, performance on this test would be the best possible indicator for someone's overall performance on all the questions in the Rust Book.

A brute-force method is to find the maximum among all subsets of size $N$. For our dataset with a decently-optimized Rust implementation, the largest $N$ that finishes in under a day on a 2021 M1 Macbook Pro is $N = 5$.
